### Задача 1.3: Error Handling (Неделя 2-3, 13-27 июля)

**Что делаем:** Система обработки ошибок — пользователь видит понятное сообщение, а не "Internal Server Error".

**Что Марк делает:**
- Создаёт `backend/exceptions.py` — кастомные exception classes с кодами
- Добавляет `ExceptionHandler` в FastAPI для каждого типа ошибки
- Добавляет graceful degradation: если DeepSeek API упал, перевод = оригинальный текст + уведомление "Translation temporarily unavailable"
- Добавляет retry с exponential backoff для внешних API (DeepSeek, readability)

**Что учим:**
- **FastAPI exception handlers** — `@app.exception_handler(HTTPException)` vs кастомные
- **Graceful degradation** — сервис не падает, а деградирует. Пользователь видит статью без перевода, а не white screen of death
- **Exponential backoff** — почему нельзя retry-ить мгновенно, как рассчитать задержку: `min(cap, base * 2 ** attempt)`
- **Circuit breaker pattern** — если DeepSeek падает 5 раз подряд, не дёргать его 10 минут

**Контрольная точка:**

**Ключевое:** Этот проект — **НЕ замена NewVision**. NewVision остаётся основным. Hotel Concierge — параллельный трек для изучения архитектуры enterprise-интеграций.
